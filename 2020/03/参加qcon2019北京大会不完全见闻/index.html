参加QCON2019北京大会不完全见闻

前言

首先非常感谢公司提供这么好的学习机会，可以去北京参与2019年QCON全球开发者大会。以前都是在网上观看各种峰会的演讲视频或是参加一些小规模的技术会议，参加QCON这种大型的技术峰会还是第一次。

通过这一次参会经历，才知道走出去看看是多么的必要：闭门造车，埋头苦干，才发现外面已然前行甚远。

开会回来之后我一直在想，怎么去写这个分享的博文。感觉从QCON会议上收获了一些东西，又感觉什么都没学到，觉得真正能在工作中实践的部分不够多。很难说通过参加一次会议就能给工作带来多大改变，更多的是对业界新技术，技术新方向，行业新动态等方面有了一些认知，下面我会分享我参会的一些想法和思考。

关于QCON

QCON简介

国际和国内技术峰会的老大哥，目前看来是国内举办最成功和最长久的技术探讨会议

QCon是由InfoQ主办的综合性技术盛会，每年在伦敦、北京、纽约、圣保罗、上海、旧金山召开。自2007年3月份开始举办以来，已经有超万名有多年从业经验的技术人员参加过QCon大会。QCon内容源于实践并面向社区，演讲嘉宾依据热点话题，面向5年以上工作经验的技术团队负责人、架构师、工程总监、开发人员分享技术创新和实践。

官网：https://2019.qconbeijing.com/

QCON2019北京站共有100+国内外技术专家，30+演讲专题，180+会议演讲，涵盖了编程语言实战,业务架构,机器学习，实时计算，前端前沿技术,技术团队管理,技术创业,混沌工程,高可用架构等软件开发的方方面面，会议议题比较全面

一些分享

分享一：《支付宝端性能体验优化实践》参会分享

支付宝端性能体验优化实践，是一场难得的移动客户端分享（在移动端至上的今天，会议关于移动端分享较少，是我比较困惑的），给了我不一样的视角和收获。

会议视频地址：https://time.geekbang.org/course/detail/177-94041

关于移动客户端性能优化，各个公司都有自己不同的优化方向和性能技术指标。大部分公司都是以性能指标来作为优化方向的，但支付宝是以业务价值为体验优化点，通过梳理用户核心使用流程，将核心流程单独拿出来作为用户体验模型，然后进行深挖，一点一点提升全路径的用户使用体验。

会议主要内容：

  • 深度解读用户体验
  • 快速启动的优化实践
  • 扫一扫极致优化
  • 持续优化的性能体系

关于深度解读用户体验

从业务场景和用户构成两方面的分析，由此带来用户体验优化需要面临的多方面挑战。在满足业务高速发展的同时，如何去提升用户体验，一直是一个比较难的话题。支付宝通过冷启动，高并发和长尾多等方面深度解读性能问题，找出解决问题的关键策略。其中技术提升和产品优化是我们可以借鉴的部分。

关于快速启动的优化实践

核心思路是找到启动的真正关键路径，使用各种方法，不只是技术也包括产品配合，避免阻塞。其实这个思路也适用于其他功能优化。

关于扫一扫极致优化

比较详细介绍了支付宝关于扫一扫极致优化的前后历程。斗鱼app也有扫一扫这个功能，虽然非核心功能点，但也可以参考优化方案进行优化

关于持续优化的性能体系

分享了构建体系化的客户端性能持续优化的架构，我认为比较重要是建设性能优化基础设施。现在我们移动端真机机房，每天在跑的稳定性测试和兼容性测试，APM监控等方面持续优化，与产品运营用户协调配合，慢慢去构建我们斗鱼的客户端性能优化体系。支付宝的性能优化体系，我们有很多可以借鉴的部分。

分享二：《从 0 到 1 搭建机器学习系统：基于小红书的个性化推荐应用为例》参会分享

要问最近几年讨论最火的几个话题，肯定会包括人工智能和机器学习。机器学习已经在很多公司进行了业务应用，我之前只是看过机器学习的一些专题文章和新闻，对机器学习了解不算太多。

刚好小红书关于机器学习的演讲从标题和介绍上吸引了我，就选择去听一下这个专题，会前我也查阅了一些资料，结果听了会议之后，才发现自己还是太年轻。会议内容对于机器学习领域的人来说可能比较好理解，对于我这种菜鸟还是太硬核了一点。虽然演讲内容从0到1的介绍了小红书搭建机器学习系统的流程，但是太多机器学习的相关词汇对于我来说，比较难理解，从而无法流畅和深入的理解演讲内容。这是比较遗憾的部分，

通过参会学习，我知道了对于机器学习如何去技术选型，如何从数据流到特征，再到模型，怎么从0到1去搭建机器学习模型有了一个初步的认知。

会议主要内容：

  • 从 0 到 1 搭建机器学习系统：技术选型，从数据流，特征到模型；
  • 机器学习的世界里技术产品应该没有边界：从产品角度入手选取合适的学习目标以及Value Model；
  • 没有数据就没有机器学习：机器学习不是黑盒，人人都要能用和会用数据。

讲师在会议提到了三个比较有意思的点，我觉得比较有意思，在这里可以给大家分享一下：

  • 技术到底有没有价值观

技术到底有价值观？也是大家一直在讨论的话题。我们是把代码和机器给出的内容直接给用户还是加上人工干预之后再给用户？这里举了一个例子，小红书上比较恶心的图片信息往往点击率很高（人的好奇心导致），如果根据机器学习无人工干预的话，就会直接推给用户，这样会带上不好的用户体验，虽然产品点击率上去了，但是实际评价和收益不会好

  • 人人都是产品经理

研发人员在参与产品需求的时候，是只是完成产品研发，而不去参与产品讨论和思考，做的对不对，用户喜不喜欢等方面一点不去考虑？技术和产品是没有界限的，技术应该有自己的思考

  • 自己要用自己的产品，自己吃自己的狗粮，自己吃自己的狗屎

自己要做自己产品的用户

分享三：《云原生架构下的混沌工程实践》参会分享

讲真，混沌工程这个概念和学科我还是第一次听说，参会之前我也查阅了一些资料,发现混沌工程已经发展有10年了。 有趣的是百度百科上居然没有混沌工程这个词条，然后我在维基百科上找到了关于混沌工程的介绍。

混沌工程的概念：在软件开发中，给定软件系统在保证足够的服务质量的同时容忍故障的能力 - 通常被概括为弹性 - 通常被指定为要求。 但是，由于诸如期限短或缺乏对该领域的了解等因素，开发团队往往无法满足此要求。 混沌工程是一种满足弹性要求的技术。混沌工程可用于实现弹性：基建失败，网络故障，应用失败。

会议主要内容：

  • 混沌工程三连问

    混沌工程是什么？
为什么要实施混沌工程？
怎么实施混沌工程？

  • 落地案例介绍

  • 未来展望

会议视频链接：https://time.geekbang.org/course/detail/177-94665

虽然这个会议有产品推广的嫌疑，但是也从云迁移、云就绪、云原生三个角度入手介绍实施混沌工程的一些方法原则和技术工具。可以让我们了解混沌工程领域的演进历史和核心原则，了解如何借助混沌工程技术，构建容错性好、弹性可扩展的云原生系统，了解开源混沌工程测试框架 ChaosBlade 的设计思路及技术实现。

大部分第一次听说和初步了解混沌工程后，会有这么一个疑问：混沌工程不是就是破坏性测试吗？其实不然，两者还是有很大差别的。虽然混沌工程跟测试通常都会共用很多测试工具的，比如都会使用错误注入工具，但混沌工程是通过实现对系统有更新的认知，而测试则是使用特定方式对某一块进行特定测试。在传统测试下，我们可以写一个断言，我们给定特定的条件，产生一个特定的输出，如果不满足断言条件，测试就会报错。但混沌工程是试验，而试验会有怎样的新信息生成，我们是不确定的。

会议上也分享了混沌工程在双11活动里面的应用，从“故障驱动”到“故障”驱动。前者的故障驱动是指故障发生后我们去修复系统和完善系统，很被动的处理故障，后者的故障驱动是指主动的拥抱故障，在线下环境和线上环境模拟实时历史故障，实施混沌工程(严格遵守混沌工程实施原则)，提升系统的稳定性和扩展性。

通过会议来看，在公司实施混沌工程需要很大的人力，物力和时间周期，但会议内容中的故障驱动这部分我们是可以借鉴的，在斗鱼，每周和每个月都有各种各样故障，如果将这些故障全部进行抽象，然后在新系统上线之前，有这么一个系统或者工具将历史故障在新系统上全部演练一次，修复可能历史故障问题后再上线，是否系统的稳定性会更好一些呢？

分享四：大型产品开发中的逻辑推理

这个专题也是这次QCON大会中唯一一个产品专题。讲师说话很直接，内容简单明了，推荐大家看一下视频。

会议主要内容：

  • 组织架构
  • 产品逻辑推理
  • 产品需求

讲师介绍了大型产品开发中的完美组织架构应该是怎么样的，项目中各个角色的职责分布，产品功能的逻辑推理五步法，大型产品开发中的功能定义以及用户故事划分，产品经理如何获取有效的反馈，大型产品如何验证用户的需求。

会议视频链接：https://time.geekbang.org/course/detail/177-94761

总结：

  • 做每件事都当成一个产品来做，搞清楚产品的目标用户，只有知道目标用户，才能知道用户的痛点，才能找到做事情的方向和目标
  • 当我看到PPT上那个组织架构图时，大部分人都笑了，估计大家发现描述的好像就是自己的公司，我发现我们公司也是如此。虽然讲师描述的组织架构和责任细分图我认为比较完美，我们公司想要达到，需要很长的路要走，但是有些我们是可以借鉴慢慢去改变的。
  • 当老师讲到软件开发中哪一项成本最高？大部分脱口而出：沟通。的确，现在沟通成为困扰斗鱼大部分研发同学的一个大问题（尤其是跨业务线，跨部门沟通）。如何有效沟通是大家都需要学习和掌握的技能。讲师中间还调侃到为啥会996，因为白天要沟通，晚上要写代码啊。
  • 关于逻辑推理方法，经过讲师一讲解，好多我之前不能理解的事情好像也能解释通了。后面我也要慢慢研究和学习更多逻辑推理的方法。
  • 讲师推荐了很多书，少部分书我已有了，没有的部分我也会买回来看完。

写在最后

一些思考

个人想法，勿喷

这一次QCON大会也是QCON十周年，这十年也是软件行业高速发展的十年，软件开发在十年间也经历了很大的变化，各种新语言和新技术，新领域等不断涌现，那软件测试呢？从最开始手工测试到自动化测试，再到精准测试，AI测试等，感觉软件测试行业有一些停滞不前，并没有随着软件行业的发展而发展，几年前我们在讨论自动化测试，现在我们还是在讨论自动化测试，这是为什么呢？值得我们是思考一下
